## 软件开发规范
### JAVA 开发规范
我们一致按照业内标准[https://github.com/google/styleguide]配置自己的ide，比如
1. import中不能有通配符。
2. 重载方法应该放在一起连续出现。
3. 即使代码块中没有内容，或者只有一行代码，也必须使用大括号。
    * 2空格缩进。
    * 列宽是80或100个字符。
    * 不能使用类似C语言的数组变量声明。
    * switch语句中，必须包含default语句。
    * 修饰符的顺序应按照Java语言规范推荐的顺序。
    * 常量命名应使用CONSTANT_CASE格式（译注：所有字母大写，单词使用下划线分隔）注意，所有常量都必须是static final成员，但并不是所有的static final成员都是常量。
    * 类名和变量名用camel case，类名首字母大写,变量名首字母小写
4. 表结构设计
  - 必须有主键
  - 枚举值尽量用整型值
  - 依赖关系理清后按照面向对象的理念设计表结构
  

### 项目开发约定
1. 尽量提供swagger 来给前端开发便于提欧式
2. 区分数据库直接耦合的dao 和 前端交互用的vo
3. mapper 的拓展尽量避免放在自动生成的mapper中,避免影响数据库修改, 复杂查询要避免笛卡尔积情况。
4. git 分支开发原则：
 - 主干开发 -master 是上面正在开发的代码
 - 分支发布 -prod 分支是可以发布到生产环境的


### 项目发布约定
1. master开发自测或者单元测试后发布到测试环境
2. 测试环境测试通过后才可以发布到生产环境
3. 上测试环境或者上生产环境都用jenkins对应的Job来实现。
   - 测试环境前后端发布jenkins job
   
   http://10.20.100.245:8080/job/zcc-backend-test/
   http://10.20.100.245:8080/job/zcc-frontend-test/
    - 生产环境前后端发布jenkins job
    
    http://10.20.100.245:8080/job/zcc-backend-prod/
    http://10.20.100.245:8080/job/zcc-frontend-prod/
  
  
### 项目开发管理约定
1. 需求尽量在禅道系统中
2. 大需求可以拆分成小需求按照优先级逐步完成
3. 在项目启动初期不明确的地方用每日站会来加快沟通频率和响应速度
